# 러브버그 출몰·민원 예측 서비스 — 설계 문서

> 서울시 자치구별 러브버그(붉은등우단털파리) 출몰 및 민원 증가 가능성을 예측하고,
> 위험 지역과 그 **이유**를 지도에서 보여주는 AI 웹 서비스.

---

## 1. 프로젝트 개요

### 1.1 한 줄 정의
"오늘/내일 서울 어느 자치구에서 러브버그 민원이 늘어날지, 그리고 **왜** 그런지를 지도로 보여주는 서비스"

### 1.2 핵심 가치 (차별점)
1. **예측**: 단순 현황이 아니라 향후 출몰·민원 증가 *가능성*을 점수화
2. **설명가능성(XAI)**: "이 구가 위험한 이유"를 변수 기여도로 제시 → 신뢰·실용성
3. **지도 시각화**: 자치구 단위 색칠(choropleth)로 직관적 전달

### 1.3 대상 사용자
- 일반 시민(외출·방역 참고)
- 자치구 환경/방역 담당 부서(선제 방역 의사결정)

## 2. 도메인 지식 (모델 설계의 근거)

러브버그(붉은등우단털파리, *Plecia longforceps*) 출몰 특성:

| 요인 | 영향 | 데이터화 방법 |
|---|---|---|
| 시기 | 6월 말~7월 초 집중 (연 1~2주 피크) | 일자/주차 계절성 변수 |
| 기온 | 고온(약 25~30℃)에서 활발 | 일 최고/평균기온 |
| 습도 | 다습 환경 선호 | 상대습도 |
| 강수 | 비 온 직후 1~3일 급증 | 강수 후 경과일, 누적강수 |
| 지형 | 산·녹지 인근에서 대량 발생 | 자치구 녹지/산림 비율 |
| 위치 | 서울 북서부(은평·서대문·마포) 집중 | 자치구 고정 가중치 |
| 인구 | 유동인구 많을수록 민원 多 | 생활인구(민원 보정) |

> 이 표가 곧 **특징(feature) 설계서**가 됩니다.

## 3. 데이터 소스

| # | 데이터 | 출처 | 형식 | 비고 |
|---|---|---|---|---|
| D1 | 날씨(기온·습도·강수) | 기상청 단기예보/관측 API (data.go.kr) | JSON | 예측 핵심 입력. 인증키 필요 |
| D2 | 민원 데이터 | 서울 열린데이터광장 / 국민신문고 / 120다산콜 | CSV/JSON | 라벨(정답). 확보 난이도 높음 |
| D3 | 자치구 경계 | VWorld / GitHub 공개 GeoJSON | GeoJSON | 지도 choropleth용 |
| D4 | 녹지·산림 비율 | 서울 열린데이터광장 | CSV | 자치구 고정 가중치 |
| D5 | 생활인구 | 서울 열린데이터광장 | CSV | 민원 발생 보정(선택) |

### 데이터 확보 리스크
- **D2(민원)가 가장 큰 리스크**: 러브버그 한정 민원 시계열을 공개 데이터로 충분히 얻기 어려울 수 있음.
- 대응: 1단계는 라벨 없이 **규칙 기반 점수**로 시작 → 데이터 모이면 ML 전환(2단계).

│ (배치/크론)  │   │ 자치구·일자별 │   │ 점수/ML+XAI │   │ FastAPI  │   │ Next.js+Leaf │
└─────────────┘   └──────────────┘   └─────────────┘   └──────────┘   └──────────────┘
       │                  │                  │                              │
   기상청 API        pandas 집계        규칙/LightGBM                 자치구 색칠 +
   GeoJSON/CSV                          + SHAP 기여도                 "위험 이유" 패널

### 4.1 기술 스택 (확정)
- **프론트엔드**: Next.js (React) + TypeScript + Leaflet (무료 OSM 타일, API 키 불필요)
- **백엔드/모델**: Python + FastAPI + pandas + (2단계) LightGBM + SHAP
- **데이터 저장**: MVP는 JSON/SQLite → 확장 시 PostgreSQL + PostGIS
- **배치**: Python 스크립트 + cron(또는 GitHub Actions 스케줄)

## 5. 예측 모델 설계

### 5.1 1단계 — 규칙 기반 위험점수 (MVP, 라벨 불필요)

#### 기본 구조 (가중 앙상블)
3개 점수의 가중합으로 자치구별 위험도를 산출한다.

```
러브버그 위험도 = 0.45 · 과거 민원 점수      (공간 baseline: 어느 구가 위험한가)
              + 0.35 · 날씨 위험 점수       (시간 강도: 이번 주가 얼마나 심한가)
              + 0.20 · 지역 환경 점수       (녹지·하천·공원 등 서식 조건)
```
- 각 점수를 0~100으로 환산 후 가중합
- **역할 분담 관점**: 과거 민원·환경(65%)은 *"어느 구"*를 결정하고, 날씨(35%)는 *"이번 시즌/주가 얼마나 심한지"*를 결정한다.
- **장점**: 라벨(정답) 없이도 즉시 지도 데모 가능, 각 항이 곧 "위험 이유"가 됨.

#### 각 점수 정의
- **과거 민원 점수**: `lovebug_words`(러브버그, 러브 버그, 러브벌레, 사랑벌레, 붉은등우단털파리)로 민원 필터 → 자치구별 과거(예: 24·25년 6월) 민원수 집계 → 정규화.
- **날씨 위험 점수**: 단순 현재 강수량이 아니라 **강수 후 경과일 + 기온 + 습도** 조합으로 산출(아래 5.1.1 참고).
- **지역 환경 점수**: 녹지 면적·공원 면적·하천 인접성 구간 점수화. 공원/녹지 중복(다중공선성) 정리 후 합산.

#### 5.1.1 설계 보정 사항 (정확도·신뢰도 향상)
초기 단순 설계 대비 아래 보정을 반영한다.

1. **과거 재현 방지(역할 분담)**: 과거 민원 45% + 환경 20% = 65%가 고정값이라 순위가 매년 거의 불변 → 위 "공간 baseline vs 시간 강도" 관점으로 날씨의 변동 역할을 명확히 한다.
2. **민원수 인구/면적 보정** ⚠️ *중요*: 인구·유동인구가 많은 구는 민원이 자동으로 부풀려짐. **인구(또는 면적)당 민원수**로 보정해 실제 출몰 밀도에 가깝게 한다.
3. **정규화 안정화**: "최대 구 = 100" 상대평가는 한 구 폭증 시 나머지가 과소평가되고 연도 간 비교가 깨짐. 분포가 한쪽으로 쏠리면 **log 변환 후 정규화**.
4. **강수 시차(lag) 반영** ⚠️ *중요*: 러브버그는 비 오는 날 활동↓, **비 온 뒤 1~3일에 급증**. 현재 강수량↑ = 위험↑로 두면 방향이 거꾸로 → **"강수 후 경과일 + 고온(25~30℃) + 다습"** 조합으로 *비 온 다음 날 고온다습 = 최고 위험* 을 표현.
5. **계절성(시기) 가중치**: 6월 말~7월 초 피크. 스코어에 시즌 가중치를 곱해 시즌 밖(7월 중순 이후·겨울)에는 전체 점수를 낮춘다. (피크 주간 ≈ 1.0, 시즌 밖 ≈ 0.x)
6. **환경 변수 중복 정리**: 공원면적·녹지면적은 상관이 높음(다중공선성) → 하나로 합치거나 설명력 높은 쪽 위주로.
7. **검증 절차**: 2024 데이터로 가중치를 잡고 **2025로 검증**(예: 상위 5개 위험구 적중률)해 신뢰도 확보.

#### 보정 반영 최종 식
```
risk(g, d) = 시즌가중치(d) ×
    [ 0.45 · 과거민원점수(g)   ← 인구/면적 보정 + log 정규화
    + 0.35 · 날씨점수(g, d)    ← 강수 후 경과일 + 기온 + 습도 (단순 강수량 X)
    + 0.20 · 환경점수(g)       ← 녹지·하천, 공원/녹지 중복 정리 ]
```
- 우선순위: **②인구 보정**과 **④강수 시차**를 먼저 반영하면 "예측"으로서의 설득력이 가장 크게 향상됨.

### 5.2 2단계 — 머신러닝 (라벨 확보 후)
- **문제 유형**: 회귀(자치구·일자별 민원 건수/증가율 예측) 또는 분류(위험 등급)
- **모델**: LightGBM/XGBoost (소량·표형 데이터에 강함)
- **특징**: §2 표의 변수 + 시차(lag) 변수(전일 기온, 누적강수 등)
- **검증**: 시계열 분할(시간 순서 유지, TimeSeriesSplit), 누수 방지
- **지표**: MAE/RMSE(회귀), F1/AUC(분류)

### 5.3 설명가능성(XAI) — 핵심 기능
- 1단계: 각 가중항의 기여도를 그대로 표시
- 2단계: **SHAP**로 자치구별 예측 근거 산출
- UI 문구 예: *"은평구 위험 ↑ — 어제 강수 + 오늘 29℃·습도 82% + 녹지 35%"*

## 6. API 설계 (FastAPI)

| 메서드 | 경로 | 설명 | 응답(요약) |
|---|---|---|---|
| GET | `/api/risk?date=YYYY-MM-DD` | 전 자치구 위험점수 | `[{gu, score, level, reasons[]}]` |
| GET | `/api/risk/{gu}?date=` | 특정 자치구 상세 | 점수 + 변수별 기여도 |
| GET | `/api/geojson` | 자치구 경계 | GeoJSON |
| GET | `/api/health` | 헬스체크 | `{status}` |

응답 예시:
```json
{
  "gu": "은평구",
  "date": "2026-06-28",
  "score": 87,
  "level": "위험",
  "reasons": [
    {"factor": "강수 후 2일", "contribution": 0.30},
    {"factor": "기온 29℃",   "contribution": 0.25},
    {"factor": "녹지비율 35%","contribution": 0.20}
  ]
}
```

## 7. 프론트엔드(UI) 설계

- **메인 지도**: 서울 자치구 choropleth (위험도별 색: 초록→노랑→빨강)
- **범례(Legend)**: 점수 구간별 색상 안내
- **상세 패널**: 구 클릭 시 점수 + "위험 이유" 기여도 막대/리스트
- **날짜 선택**: 오늘/내일 등 예측 날짜 토글
- **(확장)** 시계열 추이 미니 차트, 자치구 랭킹 리스트

라이브러리: `react-leaflet`, 무료 OSM 타일(키 불필요), GeoJSON 색칠은 `style` 콜백으로 점수 매핑.
